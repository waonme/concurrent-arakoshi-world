# エージェント引き継ぎドキュメント（agent-handoff）

このディレクトリは、複数エージェント間で作業を分担するための運用場所です。  
目的は**再現性・競合回避・完了条件の明文化**です。

## 置き場

- `README.md`（このファイル）: 運用ルール
- `AGENT_BRIEF.md`: 実装依頼の最短要件（固定版）
- `concurrent-arakoshi-world.md`: 仕様原本（最新）
- `experimental-requirements.md`: 実験改変の最終要件（優先参照）
- `progress-map.md`: フェーズ進捗の現状一覧
- `phase-*-entrypoint.md`: フェーズ単位での依頼テンプレ

## 参照優先度

1. `concurrent-arakoshi-world.md`（仕様）
2. `phase-roadmap-full.md`（実行指示）
3. `experimental-requirements.md`（実験要件）
4. `AGENT_BRIEF.md`（実装の最短要件）
5. `progress-map.md`（進捗）

## 運用ルール

1. 1人のエージェントは、原則として1つのフェーズを担当する。
2. 同一ファイルは同時編集を避ける。必要な場合は事前に責任範囲を明記する。
3. 仕様変更がある場合は、依頼と結果のどちらにも必ず「DoD（完了条件）」を付ける。
4. 既存挙動への影響は、`最小差分`を優先する。
5. 失敗時は原因を分離して報告し、再試行手順を必ず記載する。

## タスク委譲フォーマット

### 依頼メッセージ最小要件

- 依頼目的（何を何を実現するか）
- 対象ファイル（必須。触ってよい/触らないを明記）
- 依存前提（利用中の Hook や Context）
- 非機能制約（例：既存 API を壊さない、pinned/marked順）
- 完了条件（DoD）
- 検証手順（手動でも可）
- リスク・想定外ケース

### 受け取り側の報告フォーマット

- 変更ファイル
- 変更要約
- 残課題（あれば）
- テスト/確認結果
- 追加で必要な調整

## 命名ルール

- `phase-xx-*.md` は依頼・更新用。`xx` は 2桁番号
- 進捗更新は日時を明記
- TODO は「未着手/一部/完了」で統一

## ドキュメント更新ルール

- 要件変更や設計補足が入った場合は、まず `concurrent-arakoshi-world.md` を更新し、必要なら `phase-roadmap-full.md` を更新する。  
- 状態遷移（完了/一部/未着手）は、同日に `progress-map.md` へ反映する。
